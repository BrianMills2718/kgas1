# Proposal Positioning

The local `CLAUDE.md` is an unusually explicit proposal-writing contract. It says the research should be framed as feasibility testing, baseline establishment, proof-of-concept infrastructure, and methodological contribution. It also says not to claim perfect accuracy, offline-behavior prediction, production-system readiness, or theory generation as completed work. [2]

The same guidance defines the three-essay structure: Essay 1 explores how to maximize value from LLMs for computational social science; Essay 2 covers KGAS implementation; Essay 3 demonstrates the system with a COVID dataset. It also insists that graph, table, and vector are not three parallel analyses but a theory-guided cross-modal data flow where outputs from one modality become inputs to another. [2]

The main proposal draft headings preserve the proposed dissertation arc: introduction and research context, research questions and contributions, theoretical framework, system architecture, Essay 1 theory integration, Essay 2 system development, Essay 3 analysis/application, validation strategy, and timeline/limitations. [8]

# Critique Incorporation

`organized_critiques.md` begins with an academic-integrity warning and requires that capability claims use designed/proposed framing rather than functional or production-ready framing. It also shifts the validation center toward construct estimates: whether KGAS can extract construct estimates from COVID conspiracy discourse that correlate with validated psychological self-reports. [3]

The critique file calls for removing UFO material, cloud SKU details, universal-platform claims, SME validation, tool-count details, and production-deployment language. It also requires terminology changes such as "construct estimate" instead of "computational proxy" or "construct score." [3]

`critique_satisfaction_checklist.md` claims the rewritten proposal satisfied the 100+ critique items, eight technical questions, academic integrity safeguards, annex structure, citations, COVID/Kunst dataset details, and HSPC considerations. This is a self-assessment artifact and should be treated as rewrite-completion evidence, not external approval. [9]

# Uncertainty Framework

`COMPREHENSIVE_UNCERTAINTY_FRAMEWORK2.md` defines a human-interpretable uncertainty design spanning source credibility, cross-source coherence, temporal relevance, methodological limitations, technical extraction/entity recognition, construct validity, theory-data fit, evidence sufficiency, and final claim strength. It maps confidence to Intelligence Community-style probability bands to avoid false precision. [10]

`meta_schema_v14_notes.md` adds a key design insight: different uncertainty combinations answer different questions, so the system should preserve multiple dimensions rather than force a single combined score. It highlights operationalization clarity, parameter uncertainty ranges, multi-dimensional uncertainty, and DAG propagation as high-priority schema enhancements. [11]

The IC integration planning set contains an important contradiction/tension. Its executive summary describes KGAS as an autonomous LLM research system and proposes a three-level uncertainty framework for computational confidence, theory-driven confidence, and agent decision support. But its critical-insights summary says the current confidence approach was fundamentally broken because CERQual was being applied at the wrong level and because evidence aggregation, claim formation, contradiction detection, and sufficiency assessment were missing. [6] [12]

# Validation Strategy

The actual planned validation matrix prioritizes practical tests: HotPotQA multi-hop retrieval, COVID construct correlations using the Kunst dataset, hand-coded theory comparison, inter-LLM reliability, theory replication, Mechanical Turk simple tasks, optional time-series prediction, multi-resolution consistency, cross-modal capability comparison, sentiment baseline checks, and paper-plus-dataset replication. [7]

The broader validation matrix maps 14 uncertainty dimensions to validation methods and records that only part of the uncertainty framework was planned for near-term validation. It emphasizes no predetermined targets, real data, mixed methods, system validation rather than theory-truth validation, and feasible scope. [7]

# Claim Discipline

This subtree is one of the clearest examples of why the thesis record needs a wiki rather than a flattened status summary. The proposal guidance and critique documents say to avoid functional/production claims, while adjacent capability and IC-integration documents contain stronger language such as "100% functional", "production-ready", and "autonomous research system." The durable synthesis is not that one side is simply true; it is that the proposal was actively trying to discipline claims during a period when the broader KGAS materials still mixed design, proof-of-concept, experimental validation, and production language. [2] [3] [6]
