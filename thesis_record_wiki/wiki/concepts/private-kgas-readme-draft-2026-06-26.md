# Status

This is a private README-style draft. It should not replace the repository README, be copied into a public repository, or be published without a separate export/privacy review.

# KGAS

KGAS is a preserved research lineage exploring how GraphRAG-style systems could become theory-aware infrastructure for computational social science.

The project asks what changes when the goal is not only to retrieve relevant text, but to operationalize social-science theory: extract constructs from papers, preserve theoretical vocabulary, choose representations that match the theory, apply schemas to discourse, and report evidence and uncertainty boundaries.

# What KGAS Explored

The central research pipeline was:

```text
academic paper
  -> theory schema
  -> model-form routing
  -> discourse application
  -> comparable outputs
  -> validation evidence
```

The important design move was to treat theories as executable research artifacts. A useful theory schema should preserve constructs, vocabulary, formal notation, algorithms, assumptions, application stages, and validation rules. It should also be honest about what kind of representation the theory needs: graph, table, vector, sequence, statistical model, hypergraph, or hybrid form.

# Why This Was Not Just Ordinary GraphRAG

The local KGAS/Digimon work grew from the GraphRAG/DIGIMON lineage, where graph-based RAG methods are decomposed into graph types and retrieval operators. KGAS pushed that frame toward theory-grounded analysis.

For KGAS, the graph was only one possible model form. The deeper question was how an AI analysis system could preserve and apply theory structure without flattening it into generic entity extraction or fluent summaries.

# What The Record Shows

The preserved record supports these bounded claims:

- KGAS produced a substantial research lineage around theory extraction, schema generation, model-form routing, cross-modal analysis, and evidence discipline.
- The thesis/proposal materials narrowed the claim from broad product ambition toward construct estimation, baseline reporting, cross-modal comparison, and governed data use.
- The current recovered runtime has bounded local proof for selected document-analysis paths, but that runtime proof is not the same as dissertation validation.
- The uncertainty work is best read as an evolving evidence-discipline program, not as one final validated uncertainty engine.
- The wiki layer is the current navigation surface over a messy but valuable historical record.

# What Not To Claim

Do not claim:

- KGAS is a finished product;
- the dissertation was completed or accepted;
- the system has been fully validated;
- all historical architecture documents describe current runtime behavior;
- ADR-029 or Comprehensive7 uncertainty is current runtime behavior without a fresh runtime check;
- the raw archive is public-ready;
- HSPC/IRB approval was located.

# Why The Private Archive Matters

The archive preserves the work history: false starts, older ambitions, narrowed proposal language, runtime repairs, validation plans, implementation evidence, generated outputs, and claim corrections. That history is not noise. It explains why the research program evolved.

The archive is intentionally private because it may include logs, local configuration, old backups, sensitive research data, and other material requiring review. Public artifacts should be derived from this record, scanned, and documented separately.

# Current Best Use

The best current use of KGAS is as a private technical and intellectual archive:

1. use the wiki to recover what the work was trying to prove;
2. use the claim-map pages to separate vision, proposal claims, validation design, runtime proof, and governance limits;
3. use the theory-schema lineage pages to identify reusable method ideas;
4. use public/export boundary pages before sharing anything externally.

# Future Direction

The safest future public-facing direction is a smaller derived demo or essay:

- a public paper or theory excerpt;
- a schema extraction walkthrough;
- one or two public-text applications;
- comparison across graph/table/vector or hybrid forms;
- explicit evidence and uncertainty labels;
- no raw archive exposure.

That would communicate the core contribution without pretending the full historical archive is a public product.

# Review Checklist Before Reuse

- Choose the actual target: private README, portfolio page, technical blog post, academic note, or demo README.
- Remove private/internal phrasing before any public use.
- Run export review before copying text into a public repository or website.
- Check that no raw archive path, credential, local config, raw dataset excerpt, or security detail has been added.
- Keep claim boundaries intact: research lineage and bounded runtime proof, not completed validation.
